# Godpowers Architecture (v1 Design Doc)

> Status: DRAFT v0.2 (research-informed)
> Authors: Godpowers Team
> Last updated: 2026-05-09

This document is the canonical design for Godpowers as a coherent product.
v0.1 through v0.3 were skeleton; this defines v0.4 through v1.0.

The design is informed by research into how mature dev tools (GitHub Actions,
Tekton, Argo, Buildkite, VSCode, Cargo, Poetry, Bazel, Nx, Helm, Terraform,
Flyway, OpenTelemetry, Git, kubectl, gh, fly, wrangler, Anthropic's Claude
Agent SDK) solve the problems Godpowers needs to solve. See
`docs/RFC/0000-research-brief.md` for citations.

---

## 1. Product Thesis

### Who it's for
Solo founders and small engineering teams using AI coding tools who want to
ship production-grade software without enterprise process.

### What problem we solve
AI coding tools produce inconsistent, generic, undertested output by default.
Without discipline, the result is "AI-slop": code that compiles and tests
pass, but architecture is wrong, security is shallow, launch copy is generic,
nobody can resume after a context flush.

### What we are
Godpowers is a **workflow operating system for AI-assisted product
development**. Slash commands inside your AI coding tool orchestrate
specialist agents in fresh contexts to produce mechanically-verified
artifacts on disk.

### Core promise
> Type one command. Get a hardened, observable, deployed product, every step
> traceable, every artifact accountable.

## 2. The Three Load-Bearing Artifacts

The entire architecture rests on three files. If you understand these, you
understand Godpowers.

| File | Role | Pattern source |
|------|------|---------------|
| `godpowers.yaml` | **Intent**: what you want | Cargo.toml, package.json, fly.toml |
| `godpowers.lock` | **Facts**: what was resolved/done | Cargo.lock, poetry.lock |
| `.godpowers/runs/<id>/events.jsonl` | **History**: what happened | OpenTelemetry traces |

Every other architectural decision falls out from how these three relate.

### `godpowers.yaml` (Intent)

Hand-edited. Reviewable. Semver-style version constraints on plugins. This is
the document a human reads to understand the project's shape.

```yaml
apiVersion: godpowers/v1
kind: Project
metadata:
  name: myproject
  description: "..."

mode: A                  # A=greenfield, B=gap-fill, C=audit
scale: medium            # trivial | small | medium | large | enterprise

plugins:
  - "@godpowers/core@^1.0.0"
  - "@godpowers/security-pack@^1.0.0"

workflow: full-arc       # references workflows/full-arc.yaml

executors:
  default: claude-code
  fallback: codex

config:
  yolo: false
  conservative: false
```

### `godpowers.lock` (Facts)

Tool-managed. Diffable (TOML for readability, not JSON). Records what was
actually resolved. Re-running with the same lock is a no-op.

```toml
# AUTO-GENERATED by godpowers. Do not edit by hand.
version = "1"
generated-at = "2026-05-09T15:00:00Z"

[godpowers]
version = "1.0.0"

[plugins]
"@godpowers/core" = { version = "1.2.3", checksum = "sha256:abc..." }
"@godpowers/security-pack" = { version = "1.0.4", checksum = "sha256:def..." }

[agents]
god-pm = { version = "1.0.0", source = "@godpowers/core" }
god-architect = { version = "1.0.0", source = "@godpowers/core" }

[runs]
last = "2026-05-09T15:00:00Z-abc123"

[artifacts]
"prd/PRD.md" = { hash = "sha256:...", agent = "god-pm@1.0.0", run = "...-abc123" }
"arch/ARCH.md" = { hash = "sha256:...", agent = "god-architect@1.0.0", run = "...-abc123" }
```

### `.godpowers/runs/<id>/events.jsonl` (History)

Append-only. Crash-safe. JSONL for grep-ability. Standard OpenTelemetry-style
spans with attributes.

```jsonl
{"trace_id":"abc","span_id":"01","ts":"...","name":"workflow.run","attrs":{"workflow":"full-arc"}}
{"trace_id":"abc","span_id":"02","parent":"01","ts":"...","name":"agent.start","attrs":{"agent":"god-pm","tier":"prd"}}
{"trace_id":"abc","span_id":"02","ts":"...","name":"tool.call","attrs":{"tool":"Write","path":"prd/PRD.md"}}
{"trace_id":"abc","span_id":"02","ts":"...","name":"agent.end","attrs":{"agent":"god-pm","status":"success"}}
{"trace_id":"abc","span_id":"03","parent":"01","ts":"...","name":"have-nots.check","attrs":{"artifact":"prd/PRD.md","passed":15,"failed":0}}
```

This is the same shape as Datadog APM, Jaeger, Honeycomb. Users can pipe to
those tools later if they want.

## 3. Workflow Definition Language

Workflows are declarative YAML in the shape of GitHub Actions + Buildkite's
dynamic-upload trick. We do NOT invent a DSL.

### Static workflow

```yaml
# workflows/full-arc.yaml
apiVersion: godpowers/v1
kind: Workflow
metadata:
  name: full-arc
  description: Idea to hardened production

on: [/god-mode]

jobs:
  prd:
    tier: 1
    uses: god-pm@^1.0.0
    with:
      template: PRD.md

  arch:
    tier: 1
    needs: prd
    uses: god-architect@^1.0.0
    with:
      template: ARCH.md

  roadmap:
    tier: 1
    needs: arch
    uses: god-roadmapper@^1.0.0

  stack:
    tier: 1
    needs: arch
    uses: god-stack-selector@^1.0.0

  repo:
    tier: 2
    needs: stack
    uses: god-repo-scaffolder@^1.0.0

  build:
    tier: 2
    needs: [roadmap, repo]
    uses: god-build-orchestrator@^1.0.0    # composite: handles waves + reviews

  deploy:
    tier: 3
    needs: build
    uses: god-deploy-engineer@^1.0.0

  observe:
    tier: 3
    needs: deploy
    uses: god-observability-engineer@^1.0.0

  harden:
    tier: 3
    needs: build
    uses: god-harden-auditor@^1.0.0
    blocks-on:
      - critical-finding: pause

  launch:
    tier: 3
    needs: harden
    uses: god-launch-strategist@^1.0.0
```

### Dynamic uploads (Buildkite pattern)

When a planner agent needs to generate steps based on what it just learned
(e.g., god-planner generating per-slice executor jobs), it can emit additional
YAML to be appended to the workflow:

```yaml
# god-planner output:
- emit-jobs:
    - id: build-slice-1.1
      tier: 2
      uses: god-executor@^1.0.0
      with:
        slice: 1.1
        plan: build/PLAN.md#slice-1.1
      review:
        stages:
          - { uses: god-spec-reviewer@^1.0.0, on-fail: rollback }
          - { uses: god-quality-reviewer@^1.0.0, on-fail: rollback }
    - id: build-slice-1.2
      ...
```

This is the natural shape for "planner generates steps, executor runs them"
which is exactly what AI agents do.

### Why this shape

- **Familiar**: developers already know GitHub Actions field names
- **Reviewable**: YAML diffs cleanly in PRs
- **Composable**: `uses:` references plugin agents like Actions does
- **Versioned**: SemVer constraints on agents (`god-pm@^1.0.0`)
- **Dynamic when needed**: planners can emit more YAML at runtime
- **No DSL invention**: pure-DSL approaches end up reinventing programming

### Five workflow primitives (Anthropic-blessed)

We ship five primitives. We do NOT invent a sixth until the first five have
been used in production.

| Primitive | What it is | Example |
|-----------|-----------|---------|
| **Prompt chaining** | Sequential decomposition with gates | PRD -> ARCH -> ROADMAP |
| **Routing** | Classifier picks downstream specialist | Scale=trivial routes to /god-fast |
| **Parallelization** | Fan out, aggregate | Roadmap + Stack run in parallel after ARCH |
| **Orchestrator-workers** | Lead agent spawns subagents in parallel | god-build-orchestrator spawns executors per slice |
| **Evaluator-optimizer** | Generator + independent evaluator in loop | god-executor + god-spec-reviewer + god-quality-reviewer |

These are Anthropic's recommended primitives for agent workflows. We adopt
them as-is.

## 4. Agent Contract (Plugin-Style)

Agents are plugins. Plugin manifest is versioned (Helm pattern).

### Manifest schema (apiVersion: godpowers/v1)

```yaml
---
apiVersion: godpowers/v1
kind: Agent
metadata:
  name: god-pm
  version: 1.0.0
  description: Senior PM persona. Writes substitution-tested PRDs.

engines:
  godpowers: "^1.0.0"

contract:
  inputs:
    - name: user-intent
      type: string
      from: godpowers.yaml#/metadata/description
    - name: project-state
      type: file
      path: .godpowers/runs/{run-id}/state.json
  outputs:
    - path: prd/PRD.md
      contract:
        template: PRD.md
        passes:
          - have-nots: [P-01, ..., P-15]
          - substitution-test
          - three-label-test

# Tools MUST be declared explicitly. No implicit-all.
tools:
  required: [Read, Write]
  optional: [WebSearch]

yolo:
  ambiguous-problem-space:
    action: pick-broader
    rationale: Narrowing later is cheaper than expanding
  domain-knowledge-gap:
    action: log-as-open-question
  requirement-conflict:
    action: pick-higher-priority

events:
  emits: [agent.start, agent.end, tool.call, artifact.created, decision.route]

activation:
  spawned-by:
    - /god-prd
    - god-build-orchestrator
---

# Body: instructions to the LLM
```

### Why these rules matter

- **`apiVersion`**: Helm's lesson. Bake schema versioning in from day one so
  we can evolve without breaking plugins.
- **`engines.godpowers`**: VSCode's lesson. Compatibility ranges, enforced
  at install.
- **`tools` required explicit**: Anthropic's rule. No implicit-all. Smoke
  test rejects missing `tools:`.
- **`activation.spawned-by`**: VSCode lazy-loading. Agent metadata is loaded
  at install, agent body only when needed.
- **`contract.outputs.contract.passes`**: enables the test runner to verify
  every agent output mechanically.

### Plugin tiers

Two flavors, like Helm:

1. **In-process**: TypeScript/JavaScript, runs inside the AI tool's runtime.
   Rich API, primary path.
2. **Subprocess**: any language, called via stdin/stdout. Narrower contract
   but language-agnostic.

For v1.0, we ship in-process only. Subprocess is post-v1.0.

## 5. Plan-Then-Apply (Two-Phase)

Terraform's only really good idea: preview before commit.

```
god plan        # Show what would happen (model calls, tool calls, files written)
god apply       # Actually do it
god run         # = plan + apply with default approval
```

`god plan` produces a `plan.yaml` showing:
- Which agents will be spawned
- Which tools each agent will call
- Which files will be written
- Estimated cost (model tokens) and duration

User reviews, then `god apply`. For autonomous mode, `god mode` skips the
manual approval but still emits the plan to events.jsonl for audit.

This is the natural shape for AI workflows: you want to see what the agent
plans to do before it does it.

## 6. Recovery: Forward-Only with Compensation

Flyway's lesson: true transactional rollback across heterogeneous systems is
impossible. Best you can do: forward-only with cheap compensation.

| Command | What it does |
|---------|-------------|
| `god log` | Show recent state-changing operations (reflog) |
| `god undo` | Revert to previous state from reflog |
| `god redo <stage>` | Re-run a stage and downstream stages |
| `god skip <stage> --reason "..."` | Explicit skip with audit |
| `god repair` | Fix detected drift between lock and disk |
| `god doctor` | Diagnose install + project + suggest fixes |

### Reflog (Git pattern)

`.godpowers/log` is append-only, parallel to `events.jsonl` but
operation-level rather than event-level:

```jsonl
{"id":"01","ts":"...","op":"run","workflow":"full-arc","status":"success"}
{"id":"02","ts":"...","op":"plugin.install","name":"@godpowers/security-pack","version":"1.0.0"}
{"id":"03","ts":"...","op":"agent.update","name":"god-pm","from":"1.0.0","to":"1.0.1"}
```

`god undo` reverts to the state before operation N, executing compensating
operations (uninstall plugin, restore previous agent version, restore previous
artifact hashes from `.trash/`).

### Trash (recoverable deletion)

Destructive operations move files to `.godpowers/.trash/<timestamp>/<path>`
rather than deleting. `god restore` recovers from trash. After 30 days,
`god gc` cleans trash (configurable).

## 7. Onboarding: One Magic Command (Fly.io Pattern)

```
godpowers init
```

This is the fly-launch / npm-init pattern.

1. Detect project type (existing code, language, frameworks)
2. Print what was detected, allow override
3. Ask 3-5 questions max
4. Generate `godpowers.yaml`
5. Run a sample workflow to prove it works
6. Print contextual next-step suggestions

Subsequent invocations:
```
godpowers run <workflow>            # Or just: godpowers run (default workflow)
godpowers plan <workflow>           # Preview
godpowers logs <run-id>             # Inspect run history
godpowers plugin add <name>         # Add an extension
godpowers agent list                # See what agents are available
```

### Noun-verb structure (gh pattern)

- `godpowers run`
- `godpowers plugin {add, list, remove, info}`
- `godpowers agent {list, info, test}`
- `godpowers workflow {list, info, test, validate}`
- `godpowers config {get, set, list}`

Every error message ends with the most likely next command (`gh` does this
well).

## 8. Slash Commands (Inside the AI Tool)

The CLI (`godpowers`) is for project-level operations. Inside the AI coding
tool, users invoke slash commands. The slash commands are thin wrappers
around `godpowers run`:

| Slash command | CLI equivalent |
|---------------|----------------|
| `/god-mode` | `godpowers run full-arc` |
| `/god-prd` | `godpowers run prd-only` |
| `/god-arch` | `godpowers run arch-only` |
| `/god-status` | `godpowers status` |
| `/god-doctor` | `godpowers doctor` |
| `/god-rollback <tier>` | `godpowers undo --to=<tier>` |

This keeps two surfaces aligned. The slash command UX inside the AI tool
mirrors the CLI for power users.

## 9. Testing Strategy: Three Layers + Record/Replay

Per research: deterministic LLM responses are impossible. The proven pattern
is record/replay + golden-file assertions.

### Three test layers

1. **Unit tests** (pure functions): workflow parser, lock resolver, schema
   validator. Fast, hermetic.
2. **Plugin contract tests**: every agent ships `__tests__/` with fixture
   inputs. Test runner verifies contract: outputs match expected, have-nots
   pass. Plugins can't publish without passing.
3. **End-to-end tests**: spawn `godpowers run` against fixture workflows in
   temp dirs. Mock model layer at SDK boundary (record real responses once,
   replay deterministically).

### `godpowers test-plugin <path>` (Nx pattern)

Plugin authors run this before publishing. The test harness:
- Loads the plugin manifest
- Runs the agent against the bundled fixtures
- Diffs actual vs expected outputs
- Fails publish on any mismatch

### Record/replay layer

`tests/replay/` holds recorded model responses keyed by request hash. When a
test runs, the replay layer intercepts model calls and returns recorded
responses. To re-record (e.g., after intentional prompt changes):
`godpowers test --update-recordings`.

## 10. Migration from v0.3 to v1.0

### Compatibility matrix

| Version | godpowers.yaml | godpowers.lock | events.jsonl | PROGRESS.md |
|---------|---------------|----------------|--------------|-------------|
| v0.3.x  | absent | absent | absent | canonical |
| v0.4.x  | optional | optional | new | canonical (back-compat) |
| v0.5.x  | required | required | required | generated (legacy view) |
| v1.0.x  | required | required | required | optional (read-only) |

### `godpowers migrate`

```
godpowers migrate
  Detected: v0.3 project (.godpowers/PROGRESS.md exists, no godpowers.yaml)
  Will migrate to: v1.0
  
  Steps:
    1. Generate godpowers.yaml from PROGRESS.md mode + scale
    2. Generate godpowers.lock from current artifact hashes
    3. Replay synthetic events.jsonl from artifact mtimes
    4. Archive PROGRESS.md as PROGRESS.md.legacy
    5. Validate against v1.0 schemas
  
  Continue? [y/N]
```

Every migration is reversible: `godpowers migrate --dry-run` shows changes;
`godpowers migrate --rollback` restores v0.3 state from `.bak` files.

## 11. Distribution Model

### Packages on npm

| Package | Contents |
|---------|----------|
| `godpowers` | Core: CLI, runtime, schema validators |
| `@godpowers/core-agents` | The 18 default specialist agents |
| `@godpowers/core-workflows` | full-arc, audit, mode-b workflows |
| `@godpowers/core-templates` | PRD, ARCH, ROADMAP, etc. |
| `@godpowers/security-pack` | Enhanced harden agents (SOC2, HIPAA, PCI) |
| `@godpowers/launch-pack` | Enhanced launch agents (Show HN, Product Hunt) |

`npm install -g godpowers` installs core. Plugins resolved via standard npm.

### Capability handshake

```yaml
# In a plugin manifest
engines:
  godpowers: ">=1.0.0 <2.0.0"
```

`godpowers plugin add` checks compatibility before installing. Mismatches
fail with clear error and suggested action.

### Release automation

- Conventional commits on main
- Bot proposes version bumps (semantic-release pattern)
- Manual approval triggers npm publish + git tag + GitHub release
- Changelog auto-generated from commits

## 12. Anti-Patterns Rejected

| Anti-pattern | Why we reject |
|--------------|--------------|
| Storing state in conversation memory | Disk-authoritative is the entire point |
| Implicit `tools` (allow-all) | Anthropic's explicit rule. Lint rejects this. |
| Tightly coupling agents to skills | Reuse impossible if 1:1 |
| Orchestrator doing the work itself | Context rot returns; reviewers can't be independent |
| Markdown-only state | Mechanical queries and drift detection require structure |
| Implicit contracts | Breaking changes silent; versioning impossible |
| Magic auto-everything | --yolo exists, but defaults are documented and logged |
| One-size-fits-all workflow | Scale-adaptive and extensions matter |
| Hidden dependencies | All gates explicit; DAG visualizable |
| Decoration emojis as quality signals | Real icons or none |
| Theater (sentences that decide nothing) | Three-label test catches this |
| Inventing a sixth workflow primitive | Use Anthropic's five first |
| Pure-imperative workflow definition | YAML + escape hatch wins (research-validated) |
| In-memory subagent context handoff | Files only. Resumable, inspectable. |
| Transactional rollback claim | Forward-only with compensation. Honest. |

## 13. Implementation Roadmap

| Version | Focus | Key deliverables |
|---------|-------|------------------|
| **v0.4** | Foundation | godpowers.yaml + .lock, events.jsonl, schema validators, godpowers init |
| **v0.5** | Workflow Runtime | Workflow YAML parser, plan/apply two-phase, dynamic uploads |
| **v0.6** | Recovery | Reflog, undo, redo, skip, repair, doctor, trash |

## 15. Mapping Current State to Target

| v0.3 today | v1.0 target |
|------------|-------------|
| `.godpowers/PROGRESS.md` (markdown) | `godpowers.yaml` + `godpowers.lock` + auto-generated PROGRESS.md |
| Implicit workflow in orchestrator | `workflows/full-arc.yaml` declarative |
| Prose agent files | Manifest YAML front matter + prose body |
| Smoke tests (structural only) | Unit + plugin contract + end-to-end with replay |
| `npx godpowers` (1 package) | Core + per-pack ecosystem on npm |
| 26 skills + 18 agents | Same surface, declarative contracts, plugin-able |
| HAVE-NOTS.md (markdown) | Same content + machine-readable index |
| Single-machine install only | npm-distributed, capability handshake |
| God-mode / God-build / etc. | Same slash commands, backed by `godpowers run` |

The migration is additive at every step. v0.4 introduces structures while
v0.3 commands keep working. v1.0 freezes the public API.

## Appendix: Why These Specific Patterns

The architecture is grounded in concrete patterns from production tools.
Quick map:

| Pattern | Source | Where it lives in Godpowers |
|---------|--------|---------------------------|
| Intent + facts split | Cargo, Poetry | godpowers.yaml + .lock |
| Hand-edited TOML lock | Cargo (vs Poetry's JSON) | .lock format choice |
| OTel trace/span/event | OpenTelemetry | events.jsonl shape |
| Workflow YAML | GitHub Actions, Tekton | workflows/*.yaml |
| Dynamic step upload | Buildkite | Planner emits jobs |
| `apiVersion` in manifest | Helm | All schemas |
| Lazy plugin activation | VSCode | activation.spawned-by |
| Explicit tools per agent | Anthropic | Agent manifest |
| Five workflow primitives | Anthropic | The only patterns we ship |
| Plan-then-apply | Terraform | godpowers plan/apply |
| Forward-only + compensation | Flyway | godpowers undo |
| Reflog | Git | .godpowers/log |
| Magic init command | Fly, npm-init | godpowers init |
| Noun-verb CLI | gh | godpowers <noun> <verb> |
| Plugin contract tests | Nx | godpowers test-plugin |
| Record/replay | proven pattern | tests/replay/ |
| Capability handshake | npm engines | engines.godpowers |
| `npm create <thing>` style | npm | godpowers new <template> |

If a design question comes up, look it up in this table first.
